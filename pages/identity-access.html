10.2 Interview Questions

Q1How would you implement MFA for a university with diverse user populations?
â–¼
Sample Answer

"I would implement a tiered MFA strategy considering different user needs:"

IT Administrators:

  • FIDO2 security keys required (phishing-resistant)
  • No SMS fallback allowed
  • Conditional Access requiring compliant device + MFA

Faculty & Staff:

  • Microsoft Authenticator with number matching (default)
  • Windows Hello on managed devices
  • FIDO2 keys offered as option

Students:

  • Microsoft Authenticator app (primary)
  • Allow TOTP as backup
  • SMS as last resort with plan to phase out

Rollout Strategy:

  • Clear communication before enforcement
  • Registration campaign with help desk support
  • Grace period for registration
  • Self-service password reset to reduce support calls
Q2Explain Privileged Identity Management (PIM) and how you would use it.
â–¼
Sample Answer

"PIM provides just-in-time privileged access to reduce the attack surface:"

Key Features:

  • Eligible vs Active: Users are eligible for roles but must activate when needed
  • Time-Limited: Activated roles expire after set duration (e.g., 8 hours)
  • Approval Workflow: High-privilege roles require manager approval
  • Justification: Users must provide reason for activation
  • MFA Required: Additional verification on activation

IBU Implementation:

  • Global Admin: Require approval, 4-hour max, full audit
  • Exchange Admin: Self-activate with MFA, 8-hour max
  • Helpdesk roles: Eligible assignment, activate as needed

"This means even if an admin's account is compromised, the attacker doesn't have standing privileges to exploit."

Risk Types:

  • User Risk: Account likely compromised (leaked credentials, anomalous activity patterns)
  • Sign-in Risk: Specific sign-in is suspicious (impossible travel, anonymous IP, malware-linked IP)

Risk Levels: Low, Medium, High

Q4How would you handle access for a departing employee?
â–¼
Sample Answer

"Employee offboarding is critical for security - a single overlooked account can be a breach vector:

Immediate Actions (Day of Departure):

  • Disable Azure AD account: Prevents all M365 and SSO access immediately
  • Revoke active sessions: Force sign-out from all devices
  • Reset password: Even though disabled, prevents reactivation abuse
  • Remove from all groups: Clears SharePoint, Teams, app access

Data Preservation:

  • Convert mailbox to shared mailbox for manager access
  • Transfer OneDrive ownership to manager or archive
  • Document what access they had (for audit purposes)
  • Place litigation hold if required by legal

Secondary Systems Check:

  • VPN access revoked
  • Badge/physical access disabled (coordinate with facilities)
  • Third-party SaaS apps (check for personal accounts)
  • Service accounts they may have created
  • Shared credentials they knew about

Automation:

Ideally, this is triggered automatically by HR ticket, not manual notification. Integration between HR system and Azure AD can automate most of this."

Q5What is Privileged Access Management (PAM) and why does it matter?
â–¼
Sample Answer

"PAM protects the 'keys to the kingdom' - admin accounts that can access everything:

Why It Matters:

  • Admin credentials are #1 target for attackers
  • One compromised admin = entire environment compromised
  • Insider threat risk - admins have excessive standing access
  • Compliance requirements (SOC 2, PCI) mandate privileged access controls

Key PAM Principles:

  • Least Privilege: Only the access needed for the specific task
  • Just-in-Time (JIT): Access granted only when needed, auto-expires
  • Separate Accounts: Admins use different account for admin tasks
  • Session Recording: Audit trail of what admins do
  • Approval Workflow: Someone approves before access granted

Microsoft PAM Options:

  • Azure AD PIM: Just-in-time elevation for Azure AD roles
  • Privileged Access Workstations (PAWs): Dedicated hardened devices for admin work
  • Emergency Access Accounts: Break-glass accounts for when PIM fails

Without PAM, admins typically have 24/7 access to everything - that's a massive attack surface."

Q6How do you handle access reviews?
â–¼
Sample Answer

"Access reviews ensure people still need the access they have - permissions accumulate over time:

Why Reviews Matter:

  • People change roles but keep old access (privilege creep)
  • Group memberships accumulate
  • Temporary access becomes permanent
  • Compliance requires periodic recertification

Azure AD Access Reviews Process:

  • Create review: Select scope (group, app, role)
  • Assign reviewers: Managers, resource owners, or self-review
  • Set cadence: Quarterly for sensitive access, annually for general
  • Reviewer action: Approve, deny, or don't know
  • Auto-apply: Remove access for denied/non-response

What to Review:

  • Monthly: Global Admin, Exchange Admin, Security Admin roles
  • Quarterly: Privileged groups, sensitive SharePoint sites
  • Semi-annually: All group memberships, application access
  • Annually: Guest accounts, external sharing

The key is making reviews easy for managers - if it's too burdensome, they rubber-stamp everything 'approved'."
