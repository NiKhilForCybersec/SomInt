Incident Response

Methodologies, playbooks, and practical approaches to security incident handling for the university environment.

 PHASE 1: PREPARATION                                                            

 • Develop IR plan and playbooks          • Configure logging and monitoring     
 • Establish IR team and contacts         • Deploy security tools (SIEM, EDR)    
 • Conduct training and exercises         • Create communication templates       
 • Define incident severity levels        • Prepare forensic toolkit             



 PHASE 2: DETECTION & ANALYSIS                                                   

 • Monitor security alerts                • Determine incident scope             
 • Correlate events from multiple sources • Identify affected systems            
 • Validate and triage incidents          • Document initial findings            
 • Classify severity and priority         • Preserve evidence                    



 PHASE 3: CONTAINMENT, ERADICATION & RECOVERY                                    

 CONTAINMENT:           ERADICATION:           RECOVERY:                         
 • Isolate systems      • Remove malware       • Restore from backups            
 • Block IPs/domains    • Close vulnerabilities• Rebuild if needed               
 • Disable accounts     • Reset credentials    • Verify system integrity         
 • Short vs long-term   • Patch systems        • Monitor for recurrence          



 PHASE 4: POST-INCIDENT ACTIVITY                                                 

 • Conduct lessons learned meeting        • Update playbooks and procedures      
 • Document incident timeline             • Improve detection capabilities       
 • Calculate impact and costs             • Train staff on findings              
 • Report to stakeholders                 • Share threat intel (if appropriate)

Critical: Never Pay Ransom Without Consultation
Payment decisions must involve executive leadership, legal counsel, and potentially law enforcement. Payment does not guarantee data recovery and may violate sanctions laws.

Step-by-Step Ransomware Response

1
Initial Detection (0-15 minutes)
  • Identify patient zero (first infected system)
  • Document ransomware variant (photograph ransom note, note file extensions)
  • Disconnect affected systems from network (unplug cable, disable WiFi)
  • DO NOT power off - preserve memory for forensics
  • Escalate to IT Director immediately
2
Containment (15-60 minutes)
  • Block lateral movement: disable SMB (port 445), RDP (3389)
  • Isolate network segments if spreading
  • Disable compromised accounts
  • Block known malicious IPs/domains at firewall
  • Preserve affected systems for forensics
3
Assessment (1-4 hours)
  • Determine scope: which systems, what data, how many users
  • Identify entry vector (phishing, RDP, vulnerability)
  • Check for data exfiltration (double extortion)
  • Verify backup integrity (not encrypted/corrupted)
  • Check No More Ransom (nomoreransom.org) for decryptors
4
Eradication & Recovery (4-48+ hours)
  • Remove malware from all systems (may require reimaging)
  • Patch the vulnerability used for entry
  • Reset all potentially compromised credentials
  • Restore from clean backups (verify before restoring)
  • Rebuild systems that cannot be cleaned
5
Post-Incident (48+ hours)
  • Report to Privacy Commissioner (PIPEDA requires 72-hour notification for breaches)
  • Notify affected individuals if personal data involved
  • Conduct root cause analysis
  • Document lessons learned
  • Update security controls to prevent recurrence

Credential Compromise Response Commands

PowerShell - Microsoft Graph 
# 1. Force password reset and revoke all sessions
Connect-MgGraph -Scopes "User.ReadWrite.All"

$userId = "compromised.user@ibu.ca"

# Revoke all refresh tokens (forces re-authentication)
Revoke-MgUserSignInSession -UserId $userId

# 2. Check for suspicious inbox rules (common persistence method)
Get-MgUserMailFolderMessageRule -UserId $userId -MailFolderId "Inbox" | 
    Where-Object { $_.Actions.ForwardTo -or $_.Actions.RedirectTo }

# 3. Check recent sign-in activity
Get-MgAuditLogSignIn -Filter "userPrincipalName eq '$userId'" -Top 50 |
    Select-Object CreatedDateTime, IpAddress, Location, Status, AppDisplayName

# 4. Check for OAuth app consents (token theft)
Get-MgUserOAuth2PermissionGrant -UserId $userId
KQL - Hunt for Compromised Account Activity 
// Hunt for suspicious activity after credential compromise
let compromisedUser = "compromised.user@ibu.ca";
let compromiseTime = datetime(2024-01-15T10:00:00Z);

// Check for data exfiltration patterns
CloudAppEvents
| where Timestamp > compromiseTime
| where AccountObjectId == compromisedUser
| where ActionType in ("FileDownloaded", "FileSyncDownloadedFull", "FileModified")
| summarize TotalFiles=count(), TotalSize=sum(tolong(RawEventData.FileSizeBytes)) 
    by bin(Timestamp, 1h), ActionType
| order by Timestamp asc
Interview Questions

3.4 Interview Questions & Answers

Q1 Walk me through how you would respond to a ransomware attack at IBU.
Why They Ask

Ransomware is one of the top threats to educational institutions. This tests your practical incident response abilities and decision-making under pressure.

Sample Answer

"I would follow a structured approach based on the NIST incident response framework:

First 15 Minutes - Initial Response:

  • Immediately isolate affected systems by disconnecting from the network - but NOT powering off (preserves forensic evidence in memory)
  • Photograph the ransom note to identify the variant
  • Escalate to the IT Director and begin incident documentation

Containment (15-60 minutes):

  • Block lateral movement by disabling SMB (port 445) and RDP (3389) at the Meraki firewall
  • Isolate the affected network segment if the ransomware is spreading
  • Disable any accounts that may be compromised
  • Block known malicious IPs and domains associated with the ransomware C2

Assessment:

  • Determine scope: how many systems, what data types (student records? financial?)
  • Identify the entry point using Defender XDR timeline and logs
  • Check if data was exfiltrated (common in modern double-extortion ransomware)
  • Verify backup integrity - ensure backups aren't encrypted

Recovery:

  • Check No More Ransom project for available decryptors
  • Restore from clean backups after confirming they're uncompromised
  • Rebuild systems that cannot be cleaned
  • Reset all credentials before bringing systems back online

Post-Incident:

  • If student data was affected, report to the Privacy Commissioner within 72 hours per PIPEDA
  • Conduct root cause analysis and implement preventive measures
  • Document lessons learned and update playbooks

Throughout the process, I would maintain clear communication with leadership, providing regular status updates and recommendations."

Key Points to Hit
  • Demonstrate structured methodology (NIST framework)
  • Show understanding of forensic preservation
  • Reference specific tools (Meraki, Defender XDR)
  • Include compliance requirements (PIPEDA)
  • Emphasize communication and documentation
Follow-up Questions
  • "How would you determine if data was exfiltrated before encryption?"
  • "Would you recommend paying the ransom? Why or why not?"
  • "How would you communicate with students and faculty during the incident?"
Q2 A faculty member reports they clicked a phishing link. What do you do?
Why They Ask

Phishing is the #1 attack vector. This tests your response to the most common security incident you'll face.

Sample Answer

"I'd first thank them for reporting it promptly - we want to encourage reporting without fear of blame. Then I'd follow our phishing response playbook:

Immediate Questions (30 seconds):

  • Did you just click the link, or did you also enter credentials?
  • Did you download or open any files?
  • What device were you using?

If credentials were entered (HIGH PRIORITY):

  • Reset their password immediately via Azure AD
  • Revoke all active sessions and refresh tokens
  • Check for malicious inbox rules (forwarding to external addresses)
  • Review Azure AD sign-in logs for unauthorized access
  • Check for OAuth app consents they didn't authorize

If they only clicked (MEDIUM PRIORITY):

  • Run a full Defender scan on their device
  • Check browser download history
  • Monitor their account for 48-72 hours

For all cases:

  • Block the sender and malicious URL organization-wide
  • Search for other recipients who received the same email
  • Report the phishing URL to Microsoft and our threat intel feeds
  • Use this as a teachable moment - schedule brief security refresher
Code Example
PowerShell - Phishing Response 
# Quick phishing response script
$user = "faculty.member@ibu.ca"

# 1. Reset password and revoke sessions
Set-MgUserPassword -UserId $user -ForceChangePasswordNextSignIn
Revoke-MgUserSignInSession -UserId $user

# 2. Check for malicious inbox rules
Get-MgUserMailFolderMessageRule -UserId $user -MailFolderId Inbox |
    Where-Object { $_.Actions.ForwardTo -ne $null }
Q3 How do you prioritize security incidents when multiple alerts fire simultaneously?
Why They Ask

Tests your ability to triage effectively - critical skill when you're the primary security resource.

Sample Answer

"I use a risk-based prioritization framework considering four factors:

1. Threat Severity:

  • Critical: Active attack, data breach in progress
  • High: Compromised privileged account, malware on server
  • Medium: Malware on workstation, policy violation
  • Low: Suspicious but unconfirmed activity

2. Asset Value:

  • Systems containing student PII get higher priority
  • Financial systems (tuition, payroll) are critical
  • Admin/privileged systems take precedence over user workstations

3. Scope of Impact:

  • Organization-wide threat > department > individual
  • Spreading threat > contained threat

4. Containability:

  • Can we quickly contain this while we handle a higher priority item?
  • Sometimes a 30-second containment action lets us focus on bigger issues

Practical Example: If I see a ransomware alert and a failed login attempt simultaneously, ransomware gets immediate attention - I might quickly block the suspicious IP (30 seconds) then focus fully on ransomware containment.

I also leverage automation - well-tuned SOAR playbooks can handle initial containment for common scenarios while I focus on complex incidents."

Q4 Describe your experience with XDR platforms for incident response.
Why They Ask

The job mentions XDR specifically - they want to confirm you can leverage Microsoft Defender XDR for threat detection and response.

Sample Answer

"I have extensive experience with Microsoft Defender XDR (formerly Microsoft 365 Defender), which provides unified visibility across endpoints, email, identity, and cloud apps.

Key XDR Capabilities I Use Daily:

1. Incident Correlation:

  • XDR automatically correlates related alerts into incidents
  • Example: Phishing email → Credential theft → Suspicious sign-in → Data access all linked as one incident
  • This dramatically reduces investigation time versus siloed tools

2. Advanced Hunting with KQL:

  • I write custom KQL queries to hunt for threats not caught by built-in detections
  • Cross-table queries correlating endpoint, identity, and email data

3. Automated Response:

  • Configure automated investigation and remediation for common threats
  • Auto-isolate compromised devices while preserving evidence
  • Auto-disable compromised accounts

4. Attack Timeline:

  • Device timeline shows exactly what happened and when
  • Critical for root cause analysis and scope assessment

For IBU, I'd leverage Defender XDR to unify security operations across the M365 environment, providing single-pane-of-glass visibility for the IT team."

Code Example - XDR Hunt Query
KQL - Cross-Domain Threat Hunt 
// Hunt: Phishing leading to credential compromise
let phishingClicks = EmailEvents
| where ThreatTypes has "Phish"
| where DeliveryAction == "Delivered"
| project RecipientEmailAddress, NetworkMessageId, Timestamp;

let suspiciousSignIns = AADSignInEventsBeta
| where RiskLevelDuringSignIn in ("high", "medium")
| project AccountUpn, Timestamp, IPAddress, RiskLevelDuringSignIn;

phishingClicks
| join kind=inner suspiciousSignIns 
    on $left.RecipientEmailAddress == $right.AccountUpn
| where suspiciousSignIns.Timestamp between 
    (phishingClicks.Timestamp .. (phishingClicks.Timestamp + 24h))
| project User=AccountUpn, PhishingTime=phishingClicks.Timestamp, 
    SignInTime=suspiciousSignIns.Timestamp, SignInIP=IPAddress, Risk=RiskLevelDuringSignIn
Q5 How do you document incidents and maintain chain of custody for potential legal proceedings?
Why They Ask

Proper documentation is critical for compliance reporting, insurance claims, and potential legal action.

Sample Answer

"Documentation and evidence handling are critical - especially given PIPEDA reporting requirements. My approach:

Incident Documentation:

  • Timeline: Chronological record of all events with timestamps
  • Actions taken: Who did what, when, and why
  • Evidence collected: Screenshots, logs, memory dumps
  • Decisions made: Rationale for key decisions
  • Communications: All internal and external communications logged

Chain of Custody:

  • Document who collected evidence, when, and how
  • Use write-blockers for disk imaging
  • Hash all collected files (SHA-256) to prove integrity
  • Store evidence in secure, access-controlled location
  • Log every access to evidence

For IBU specifically:

  • I'd use a ticketing system to track incidents with required fields
  • Create templates for common incident types
  • Ensure documentation meets PIPEDA breach notification requirements
  • Coordinate with legal counsel on evidence preservation for serious incidents
