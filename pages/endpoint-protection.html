Attack Surface Reduction

4.3 Attack Surface Reduction (ASR) Rules

Recommended ASR Rules for Education

ASR RULES - PRIORITY DEPLOYMENT
CRITICAL RULES (Deploy Immediately)


RULE: Block Office apps from creating executable content
GUID: 3b576869-a4ec-4529-8536-b80a7769e899
 Blocks: Office creating .exe, .dll, .scr files
 Impact: Low (minimal legitimate use)
 IBU Use: Prevent macro-based malware

RULE: Block Office apps from creating child processes  
GUID: d4f940ab-401b-4efc-aadc-ad5f3c50688a
 Blocks: Word/Excel spawning cmd.exe, PowerShell
 Impact: Low-Medium
 IBU Use: Stop fileless malware from Office docs

RULE: Block executable content from email client and webmail
GUID: be9ba2d9-53ea-4cdc-84e5-9b1eeee46550
 Blocks: Running executables from Outlook
 Impact: Low
 IBU Use: Prevent email-delivered malware

RULE: Block credential stealing from LSASS
GUID: 9e6c4e1f-7d60-472f-ba1a-a39ef669e4b2
 Blocks: Mimikatz-style credential dumps
 Impact: Low
 IBU Use: Protect against lateral movement

HIGH PRIORITY RULES (Deploy in Audit First)


RULE: Block process creations from PSExec and WMI commands
GUID: d1e49aac-8f56-4280-b9ba-993a6d77406c
 Blocks: Remote execution via PSExec/WMI
 Impact: Medium (may affect IT tools)
 IBU Use: Block lateral movement

RULE: Block untrusted and unsigned processes from USB
GUID: b2b3f03d-6a65-4f7b-a9c7-1c7ef74a9ba4
 Blocks: Running unsigned code from removable media
 Impact: Medium
 IBU Use: Protect lab computers from USB threats

RULE: Use advanced protection against ransomware
GUID: c1db55ab-c21a-4637-bb3f-a12568109d35
 Blocks: Suspicious ransomware behaviors
 Impact: Low
 IBU Use: Additional ransomware protection layer

Deploying ASR Rules via Intune

PowerShell - ASR Configuration 
# ASR Rule States: 0=Disabled, 1=Block, 2=Audit, 6=Warn

# View current ASR rule status
Get-MpPreference | Select-Object -ExpandProperty AttackSurfaceReductionRules_Ids

# Enable ASR rule in Block mode
Add-MpPreference -AttackSurfaceReductionRules_Ids "3b576869-a4ec-4529-8536-b80a7769e899" `
    -AttackSurfaceReductionRules_Actions 1

# Enable ASR rule in Audit mode (for testing)
Add-MpPreference -AttackSurfaceReductionRules_Ids "d4f940ab-401b-4efc-aadc-ad5f3c50688a" `
    -AttackSurfaceReductionRules_Actions 2

# Check for ASR events in Event Viewer
Get-WinEvent -LogName "Microsoft-Windows-Windows Defender/Operational" | 
    Where-Object { $_.Id -in (1121, 1122, 1125, 1126) } |
    Select-Object TimeCreated, Id, Message -First 20
Interview Questions

4.4 Interview Questions & Answers

Q1 How would you implement device management for a mix of university-owned and student personal devices?
Why They Ask

Universities have complex device ecosystems. This tests your ability to balance security with privacy and user experience for different device ownership models.

Sample Answer

"I would implement a tiered approach using Microsoft Intune that respects ownership while maintaining security:

Tier 1: University-Owned Devices (Full MDM)

  • Enroll via Windows Autopilot or Apple Business Manager
  • Full device management with compliance policies
  • BitLocker/FileVault encryption mandatory
  • Security baselines applied
  • Software deployment and updates controlled
  • Device wipe capability for offboarding

Tier 2: Student/Staff Personal Devices (MAM-only)

  • App Protection Policies (no device enrollment required)
  • Only manage university data within apps (Outlook, Teams, OneDrive)
  • Require PIN/biometric to access work apps
  • Prevent copy/paste from work to personal apps
  • Selective wipe (university data only) if needed
  • Respects privacy - cannot see personal apps or data

Conditional Access Integration:

  • Require compliant device OR app protection policy
  • Block access from unmanaged devices to sensitive data
  • Allow web access with limited functionality for unmanaged devices

This approach gives us security controls over university data while respecting that students own their devices and have privacy expectations."

Key Points to Hit
  • Different management levels for different ownership
  • MAM (Mobile Application Management) for BYOD
  • Privacy considerations for personal devices
  • Conditional Access to enforce policies
Q2 Explain how Microsoft Defender for Endpoint detects and responds to threats.
Sample Answer

"Defender for Endpoint uses multiple detection layers:

Detection Methods:

  • Signature-based: Traditional AV signatures for known malware
  • Behavioral analysis: Machine learning models detect suspicious process behavior
  • Cloud intelligence: Real-time queries to Microsoft's threat intelligence
  • Attack surface reduction: Blocks common attack techniques proactively

EDR Capabilities:

  • Process tree analysis: Track parent-child process relationships
  • File activity monitoring: Track file creation, modification, deletion
  • Network connections: Monitor outbound connections, detect C2
  • Registry changes: Track persistence mechanisms

Response Actions:

  • Automated remediation: Auto-quarantine malware, undo malicious changes
  • Device isolation: Network isolate while preserving cloud connectivity
  • Live Response: Remote shell for investigation and remediation
  • Custom detection rules: Create rules for organization-specific threats
Code Example - Threat Investigation
KQL - Hunt for Suspicious PowerShell 
// Hunt for encoded PowerShell commands (common malware technique)
DeviceProcessEvents
| where Timestamp > ago(7d)
| where FileName =~ "powershell.exe"
| where ProcessCommandLine has_any ("-enc", "-EncodedCommand", "FromBase64String")
| project Timestamp, DeviceName, AccountName, ProcessCommandLine
| order by Timestamp desc
Q3 What are Attack Surface Reduction rules and which would you prioritize for a university?
Sample Answer

"ASR rules are proactive controls that block common attack techniques before they execute. They're part of Defender for Endpoint and can be deployed via Intune or Group Policy.

Top Priority Rules for IBU:

1. Block Office macros from creating executable content - Universities receive many documents from external sources. This blocks macro malware which is extremely common.

2. Block credential stealing from LSASS - Prevents tools like Mimikatz from dumping credentials. Critical for stopping lateral movement if an attacker gets initial access.

3. Block executable content from email - Prevents running executables attached to or downloaded from emails. Students and faculty are prime phishing targets.

4. Block untrusted processes from USB - Lab computers are high risk for USB-based threats. This protects shared computers.

Deployment Strategy:

  • Start in Audit mode to identify false positives
  • Review logs for 1-2 weeks
  • Add exclusions for legitimate applications
  • Move to Block mode after validation
  • Monitor for user impact and adjust as needed
Q4 A device is detected with malware. Walk me through your response using Defender for Endpoint.
Sample Answer

"I'd use Defender for Endpoint's investigation capabilities:

1. Initial Assessment (in Security Portal):

  • Review the alert details - what was detected, severity, file path
  • Check if automated investigation already ran
  • Note the device name, user, and time of detection

2. Device Investigation:

  • Open the device page and review Timeline
  • Trace back: What process created the malicious file? How did it get there?
  • Check for lateral movement - did it spread?
  • Review network connections - any C2 communication?

3. Containment (if needed):

  • Use 'Isolate Device' to cut network access while maintaining cloud management
  • This prevents spread while allowing continued investigation

4. Remediation:

  • If auto-remediation completed - verify malware was removed
  • If manual action needed - use Live Response to investigate and clean
  • Run full AV scan to check for additional threats

5. Post-Incident:

  • Check if other devices are affected (Threat Analytics)
  • Determine root cause - user action? Vulnerability?
  • Update detection rules if needed
  • User education if phishing-related
Q5 How do you handle endpoint security for remote users and work-from-home scenarios?
Sample Answer

"Remote work requires a cloud-first security approach. For IBU:

Identity & Access:

  • MFA required for all remote access
  • Conditional Access policies: block risky sign-ins, require compliant device
  • Risk-based authentication - challenge suspicious logins

Device Security:

  • Defender for Endpoint provides protection regardless of network location
  • Cloud-delivered protection - always up-to-date signatures
  • Compliance policies enforced through Intune
  • Always-on VPN for sensitive operations (optional)

Data Protection:

  • DLP policies prevent data leakage from managed apps
  • Sensitivity labels encrypt documents automatically
  • Block downloads to unmanaged devices via Conditional Access

Network Security:

  • Defender for Endpoint web filtering works everywhere
  • DNS protection through Defender (even off-network)
  • VPN split tunneling configured to protect corporate traffic

The key is that security follows the user, not the network perimeter."
