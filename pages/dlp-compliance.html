Data Loss Prevention & Compliance

Comprehensive guide to protecting sensitive data and ensuring regulatory compliance in Canadian educational institutions.

10.1 What is Data Loss Prevention (DLP)?

DLP Fundamentals - Plain English Explanation

Simple Definition: DLP is a set of tools and processes that prevent sensitive data from leaving your organization through unauthorized channels - whether accidentally or maliciously.

Think of it like this: Imagine a security guard at a building exit who checks everyone's bags. But instead of physical items, DLP checks digital content (emails, file uploads, USB transfers) for sensitive information like credit card numbers, student records, or confidential documents.

•
Why Universities Need DLP
Universities handle massive amounts of sensitive data: student grades, financial aid information, Social Insurance Numbers, health records, research data, and payment information. A single data breach can result in regulatory fines, lawsuits, and severe reputational damage.

10.2 Microsoft Purview DLP - Deep Dive

What is Microsoft Purview?

Microsoft Purview (formerly Microsoft 365 Compliance Center) is Microsoft's unified data governance platform. It combines:

  • Data Loss Prevention: Prevent sensitive data from leaving
  • Information Protection: Classify and encrypt data
  • Data Lifecycle Management: Retain and delete data appropriately
  • Compliance Manager: Track regulatory compliance
  • eDiscovery: Find data for legal holds

Sensitive Information Types (SITs) Explained

SITs are the patterns DLP uses to detect sensitive data. They use a combination of:

  • Regular expressions: Pattern matching (e.g., 9 digits in XXX-XXX-XXX format for SIN)
  • Keywords: Context words near the pattern (e.g., "SIN", "Social Insurance")
  • Checksums: Mathematical validation (Luhn algorithm for credit cards)
  • Confidence levels: High, Medium, Low based on how many indicators match

Key Built-in Canadian SITs:

Sensitive Information TypePatternUse Case
Canada Social Insurance Number 9 digits with Luhn check Employee records, student financial aid
Canada Bank Account Number 7-12 digits with institution code Payroll, tuition refunds
Canada Driver's License Number Province-specific formats ID verification
Canada Health Service Number 10 digits (province-specific) Student health services
Canada Passport Number 2 letters + 6 digits International student records
Credit Card Number 13-16 digits with Luhn check Payment processing

Creating Custom SITs for IBU:

You might create custom SITs for:

  • Student ID: e.g., "IBU" + 7 digits
  • Grade patterns: Keywords like "final grade", "GPA", "transcript"
  • Research data codes: Project identifiers for sensitive research

10.3 Sensitivity Labels - Classification & Protection

What Are Sensitivity Labels?

Sensitivity labels are like stamps that classify documents and can automatically apply protections. Unlike DLP (which inspects content), labels travel WITH the document permanently.

10.4 PIPEDA Compliance - Canadian Privacy Law

Understanding PIPEDA

What is PIPEDA? The Personal Information Protection and Electronic Documents Act is Canada's federal privacy law governing how private-sector organizations collect, use, and disclose personal information in the course of commercial activities.

!
Important: Which Law Applies?
• Private universities (like IBU): PIPEDA applies
• Public universities in Ontario: FIPPA (Freedom of Information and Protection of Privacy Act) applies
• Both: If handling health info, PHIPA may also apply
Know which regulations apply to YOUR institution!

PIPEDA's 10 Fair Information Principles:

#PrinciplePlain EnglishIT Implementation
1 Accountability Someone must be responsible for privacy Designate Privacy Officer; document data handling
2 Identifying Purposes Tell people WHY you're collecting their data Privacy notices on forms; clear consent language
3 Consent Get permission before collecting/using data Opt-in checkboxes; consent management system
4 Limiting Collection Only collect what you actually need Audit data collection forms; minimize fields
5 Limiting Use/Disclosure/Retention Use data only for stated purposes; don't keep forever Retention policies; data lifecycle management
6 Accuracy Keep personal info accurate and current Self-service portals; regular data quality reviews
7 Safeguards Protect data with appropriate security Encryption, DLP, access controls, monitoring

9 Individual Access People can see what data you have about them Subject Access Request (SAR) process
10 Challenging Compliance People can complain and you must respond Complaint handling process; audit trails

Interview Questions

10.6 Interview Questions & Answers

Q1 How would you implement DLP to protect student data at a university?
Sample Answer

"I'd implement a phased approach using Microsoft Purview DLP:

Phase 1 - Discovery (Weeks 1-4):

  • Enable policies in audit-only mode to understand current data flows
  • Run content searches to find where sensitive data lives (SharePoint, OneDrive, email)
  • Identify high-risk sharing patterns and establish baseline metrics
  • Interview stakeholders (Registrar, Financial Aid, HR) about legitimate data sharing needs

Phase 2 - Policy Development (Weeks 5-8):

  • Create custom Sensitive Information Types for student IDs, grade patterns
  • Design policies for different sensitivity levels (SIN = block, grades = warn)
  • Define exceptions for legitimate business processes (e.g., transcript requests)
  • Build user communication and training materials

Phase 3 - Warn Mode (Weeks 9-12):

  • Deploy policies in 'warn with override' mode
  • Users see policy tips but can proceed with justification
  • Monitor override reasons to identify policy gaps or false positives
  • Provide just-in-time training when warnings appear

Phase 4 - Enforcement (Weeks 13+):

  • Move critical policies (SIN, credit cards) to block mode
  • Keep lower-sensitivity policies in warn mode
  • Establish ongoing review process for false positives
  • Report metrics to leadership monthly

This phased approach ensures we don't disrupt legitimate work while progressively improving protection."

Q2 What is the difference between DLP and sensitivity labels?
Sample Answer

"They're complementary but serve different purposes:

DLP (Data Loss Prevention):

  • Focus: Inspects content at boundaries (email, file sharing, USB)
  • Mechanism: Pattern matching - looks FOR sensitive data
  • Action: Block, warn, or audit based on content detected
  • Scope: Active only at inspection points (send email, upload file)
  • User involvement: Often invisible until triggered

Sensitivity Labels:

  • Focus: Classification that travels WITH the document
  • Mechanism: Manual or auto-applied classification
  • Action: Persistent protection (encryption, watermarks, access controls)
  • Scope: Protection follows document everywhere, even outside the organization
  • User involvement: Users often select labels manually

How They Work Together:

  • DLP can trigger auto-labeling: 'If SIN detected, apply Highly Confidential label'
  • DLP can use labels as conditions: 'Block external sharing of Restricted-labeled files'
  • Labels provide persistent protection; DLP provides boundary enforcement

Think of DLP as the security guard checking bags at the door, while labels are like a lock on the briefcase that stays locked wherever it goes."

Q3 A data breach is discovered - student records found online. Walk me through your response.
Sample Answer

"I would follow a structured incident response approach:

1. Immediate Containment (First Hour):

  • Verify the breach - confirm data is authentic, not fabricated
  • Identify and close the attack vector if still active
  • Preserve evidence - don't modify or delete logs
  • Escalate to IT Director and initiate incident response plan

2. Assessment (Hours 2-24):

  • Scope: How many records? What data fields? What time period?
  • Source: How did the breach occur? External attack? Insider? Third party?
  • RROSH Analysis: Is there 'Real Risk of Significant Harm'? (SIN = definitely yes)
  • Engage legal counsel and consider external forensics if needed

3. Notification (As Soon as Feasible):

  • Privacy Commissioner: Report via OPC breach reporting form if RROSH exists
  • Affected individuals: Direct notification with: what happened, what data exposed, what we're doing, what they should do (credit monitoring, password changes)
  • Other organizations: Notify banks if financial data; credit bureaus if identity theft risk

4. Remediation & Recovery:

  • Close vulnerability that allowed the breach
  • Reset credentials if compromised
  • Consider offering credit monitoring to affected individuals
  • Request takedown of exposed data

5. Post-Incident:

  • Conduct lessons learned review
  • Update policies and controls
  • Document everything for PIPEDA's 24-month record requirement
  • Report to board/leadership on improvements made

The key is speed on containment and transparency in communication - trying to hide a breach makes everything worse."

Q4 How do you handle DLP false positives while maintaining security?
Sample Answer

"False positives are the #1 challenge in DLP - too many and users ignore warnings entirely. Here's my approach:

Prevention - Design Policies Well:

  • Use high confidence levels (require multiple indicators, not just pattern)
  • Add context requirements (keyword proximity: 'SIN' near the number)
  • Set appropriate thresholds (don't alert on 1 credit card number; alert on 10)
  • Test extensively in audit mode before enforcement

Detection - Monitor Continuously:

  • Review override reasons weekly - patterns indicate policy issues
  • Track false positive rates per policy
  • Maintain feedback channel for users to report issues

Response - Handle Quickly:

  • When legitimate false positive reported, create exception quickly (same day if possible)
  • Use scoped exceptions (specific user/group/location) rather than broad exclusions
  • Document why exception was granted for audit purposes

Continuous Improvement:

  • Monthly policy review meetings
  • Tune policies based on actual results, not assumptions
  • Balance security with usability - a policy everyone bypasses provides zero security

The goal is to make DLP invisible for legitimate work while catching actual violations."

Q5 What's the difference between PIPEDA and FIPPA, and which applies to IBU?
Sample Answer

"Great question - this is crucial for compliance planning:

PIPEDA (Personal Information Protection and Electronic Documents Act):

  • Jurisdiction: Federal law, applies to private sector organizations
  • Scope: Collection, use, disclosure of personal information in commercial activities
  • Key features: Mandatory breach notification, consent requirements, 10 fair information principles
  • Enforcement: Privacy Commissioner of Canada; up to $100K penalties

FIPPA (Freedom of Information and Protection of Privacy Act - Ontario):

  • Jurisdiction: Provincial law, applies to Ontario public institutions
  • Scope: Public universities, hospitals, school boards, government bodies
  • Key features: Freedom of information requests, privacy protection, data residency (some interpretations require Canadian storage)
  • Enforcement: Information and Privacy Commissioner of Ontario

For IBU (Private University):

  • As a private institution, PIPEDA applies
  • This means mandatory breach notification requirements
  • Less restrictive on cloud services than FIPPA (no Canadian data residency requirement)
  • Still need strong privacy controls but more flexibility in implementation

Important: If IBU partners with public institutions or handles Ontario health data (PHIPA), additional regulations may overlap. Always verify specific requirements with legal counsel."

Q6 How would you implement Endpoint DLP for USB and printing controls?
Sample Answer

"Endpoint DLP extends protection beyond cloud services to local device actions:

Prerequisites:

  • Devices must be onboarded to Microsoft Defender for Endpoint
  • Windows 10/11 Enterprise or Education editions
  • Devices enrolled in Intune for policy deployment
  • Appropriate M365 licensing (E5 or E5 Compliance add-on)

USB/Removable Media Controls:

  • Option 1 - Block All: Simplest but may disrupt legitimate needs
  • Option 2 - Allow Approved Devices: Whitelist encrypted corporate USB drives by hardware ID
  • Option 3 - DLP Inspection: Allow USB but scan for sensitive content; block if detected
  • My recommendation: Start with audit mode to understand usage patterns

Printing Controls:

  • Block printing of documents with sensitivity labels (e.g., 'Highly Confidential')
  • Allow printing but apply watermarks automatically
  • Audit printing of sensitive content without blocking (for awareness)
  • Consider: some departments (HR, Finance) may have legitimate print needs

Other Endpoint Controls:

  • Clipboard: Block copy/paste of sensitive data to non-approved apps
  • Cloud Upload: Block uploads to personal Dropbox/Google Drive
  • Network Shares: Restrict copying sensitive files to non-approved locations

Implementation Approach:

  1. Start with audit mode for 30 days
  2. Review activity logs to identify legitimate use cases
  3. Create exception groups for users with valid needs
  4. Move to warn mode, then block for highest-sensitivity data

The key is balancing security with usability - overly restrictive controls lead to shadow IT workarounds that are even less secure."
